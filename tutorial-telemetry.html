概要

前提条件

開始前に以下が揃っていることを確認してください。

要件備考
forge CLI ≥ 0.9.0forge --version で確認。セットアップは インストール ページを参照。
Python 3.10+ または Dockerシミュレーションエンジンはローカルプロセスで動作します。クラウド依存はありません。
テレメトリ JSONL ファイル1行1JSONオブジェクト。フィールド: ts、host、数値シリーズ1件以上。スキーマは Step 1 を参照。
空きディスク ≥ 500 MBシミュレーション出力と中間状態はデフォルトで ./output/ に書き込まれます。
このチュートリアルのすべての処理はワークステーション上でローカル実行されます。テレメトリデータは ホスト外に送信されません。AlphaForge はシミュレーション実行中に外部通信を行いません。
概要

アーキテクチャ

AlphaForge は3ステージパイプラインとして動作します: 取り込み、ディスクリプター評価、 結果出力。各ステージは独立して設定・テスト可能です。 

┌─────────────────────────────────────────────────────────────────┐
│                     AlphaForge Pipeline                         │
│                                                                 │
│  ┌──────────────┐   ┌──────────────────┐   ┌────────────────┐  │
│  │  forge ingest│──▶│  forge run       │──▶│  forge         │  │
│  │              │   │  (descriptor     │   │  dashboard     │  │
│  │ JSONL / CSV  │   │   evaluation)    │   │  (result view) │  │
│  │ Parquet      │   │                  │   │                │  │
│  └──────────────┘   └──────────────────┘   └────────────────┘  │
│         │                    │                      │           │
│    ./data/series/       ./output/runs/         ./output/        │
│    (normalised store)   (per-run state)        anomalies.jsonl  │
└─────────────────────────────────────────────────────────────────┘

ディスクリプター（.yaml）は、監視するシリーズ、ローリングベースラインの計算方法、 および異常イベントを構成するしきい値条件を定義します。ディスクリプターには実行ロジックは含まれません。 AlphaForge がシミュレーション時にこれを解釈します。

Step 1 of 5

テレメトリデータの取り込み

forge ingest を使って、生のJSONLログファイルを AlphaForge の正規化時系列ストアに読み込みます。 このコマンドはタイムスタンプフィールドを検証し、数値カラムをキャストし、ホスト識別子でデータをパーティション分割します。

サンプル入力ファイル — telemetry.jsonl

{"ts":"2026-01-15T08:01:00Z","host":"web-01","cpu_pct":89.7,"mem_pct":71.3,"net_latency_ms":420.8}
{"ts":"2026-01-15T08:02:00Z","host":"web-01","cpu_pct":91.2,"mem_pct":73.0,"net_latency_ms":510.3}

{"ts":"2026-01-15T08:00:00Z","host":"db-01", "cpu_pct": 8.3,"mem_pct":82.5,"net_latency_ms":2.9}

  --format jsonl \
  --timestamp-field ts \
  --partition-field host \

Step 2 of 5

ディスクリプターの作成

ディスクリプターは、AlphaForge に監視するシリーズ、ローリングベースラインの計算方法、 および異常イベントを発生させるしきい値条件を指示する宣言型YAMLファイルです。 スクリプトやコンパイル済みコードは不要です。

ディスクリプターファイルの作成

forge strategy create \
  --template anomaly_detector \
  --id resource_spike_detector \
  --out descriptors/resource_spike_detector.yaml

生成されたファイルを開き、以下のように設定してください。各フィールドはインラインでドキュメント化されています。

descriptors/resource_spike_detector.yaml

# AlphaForge Descriptor — Server Resource Spike Detector
# Version schema: https://alforge-labs.github.io/schema/descriptor/v1
id: resource_spike_detector
version: "1.0.0"
description: >
  Detects sustained CPU, memory, and network-latency spikes in per-host
  server telemetry. Emits a structured anomaly event whenever a series
  exceeds its rolling baseline by the configured sigma multiplier for at
  least the specified minimum duration.

# ── Input ────────────────────────────────────────────────────────────────
input:
  partition_field: host          # Group observations by this field
  timestamp_field: ts

# ── Series Definitions ───────────────────────────────────────────────────
series:
  - name: cpu_pct
    unit: percent                # Informational; not used in computation
    window: 10m                  # Rolling window for baseline computation
    baseline: rolling_mean       # Options: rolling_mean | rolling_median | rolling_percentile

  - name: mem_pct
    unit: percent
    window: 10m
    baseline: rolling_mean

  - name: net_latency_ms
    unit: milliseconds
    window: 5m
    baseline: rolling_percentile
    percentile: 95               # Required when baseline: rolling_percentile

# ── Anomaly Thresholds ───────────────────────────────────────────────────
thresholds:
  cpu_pct:
    sigma: 3.0                   # Standard deviations above baseline
    min_duration: 2m             # Must persist this long to emit an event

  mem_pct:
    sigma: 2.5
    min_duration: 5m

  net_latency_ms:
    sigma: 4.0
    min_duration: 30s

# ── Output ───────────────────────────────────────────────────────────────
output:
  format: jsonl                  # Options: jsonl | csv | parquet
  path: ./output/anomalies.jsonl
  include_context_window: true   # Append pre/post-event observations to each record
  context_window: 5m             # Lookback/lookahead duration

ディスクリプターの登録

forge strategy save descriptors/resource_spike_detector.yaml

# Output:
# ✔  Registered  resource_spike_detector  (v1.0.0)
ベースラインオプションの概要:
rolling_mean — ウィンドウの算術平均。孤立したスパイクに敏感。
rolling_median — 外れ値に頑健。持続的な変化への反応は遅い。
rolling_percentile — 長尾分布（レイテンシーなど）に使用。

Step 4 of 5

シミュレーションの実行

取り込まれた時間範囲全体でシミュレーションを実行します。AlphaForge は各パーティションを反復処理し、 ローリングベースラインを計算し、各観測でしきい値を評価して、設定された出力パスに異常イベントを書き出します。 

forge run \
  --strategy resource_spike_detector \
  --data ./data/series/ \
  --from 2026-01-15T00:00:00Z \
  --to   2026-01-15T23:59:59Z \
  --output ./output/ \
  --workers 4

CLIオプション — forge run

オプション型デフォルト説明
--strategy <ID> string — 登録済みディスクリプターID、または .yaml ファイルへのパス。

--log-level <LEVEL> enum info 詳細度: debug | info | warn | error。

CLIオプション — forge ingest

オプション型デフォルト説明
--format <FMT> enum jsonl 入力フォーマット: jsonl | csv | parquet。
--timestamp-field <F> string ts タイムスタンプカラムの名前。ISO-8601またはUnixエポック（ms）を指定。

トラブルシューティング

  • ソースデータに明確なスパイクがあるにもかかわらず異常イベントが出力されない

    最も一般的な原因は、min_duration がデータ内の実際のスパイク持続時間を超えていることです。 min_duration を短くして（例: 5m → 30s）再実行してください。 また、ローリングウィンドウが観測範囲全体より広い場合は、意味のあるベースライン計算に十分な行数が残りません。 

    # 特定シリーズとパーティションのローリングベースラインを確認
forge debug baseline \
  --strategy resource_spike_detector \
  --series cpu_pct \
  --partition web-01 \
  --output ./debug/
  • ParseError: invalid timestamp で取り込みが失敗する

    AlphaForge はISO-8601 UTCタイムスタンプ（例: 2026-01-15T08:00:00Z）または Unixエポック（ミリ秒）を期待します。ソースが異なるフォーマットを使用している場合は、取り込む前に変換してください。 

    # jq を使ってエポック秒をISO-8601に変換
jq 'del(.ts) | .ts = (.epoch_s | todate)' raw.jsonl > telemetry.jsonl
  • 大きなJSONLファイルでメモリ消費が多い

    取り込む前にParquetに変換してください。Parquetはカラム型でプレディケートプッシュダウンをサポートしており、 広いスキーマのメモリフットプリントを大幅に削減します。 

    # DuckDB を使ってJSONLをParquetに変換（uv経由でインストール不要）
uvx duckdb -c "
  COPY (SELECT * FROM read_json_auto('telemetry.jsonl'))
  TO 'telemetry.parquet' (FORMAT PARQUET);
"

forge ingest telemetry.parquet \
  --format parquet \
  --partition-field host \
  --series cpu_pct mem_pct net_latency_ms
  • 大規模データセットでシミュレーションが遅い

    --workers を利用可能なCPUコア数に合わせて増やしてください。各パーティションは独立して処理されるため、 並列性はパーティション数まで線形にスケールします。 

    forge run \
  --strategy resource_spike_detector \
  --workers $(nproc) \    # Linux
  --output ./output/

# macOS の場合
forge run --strategy resource_spike_detector --workers $(sysctl -n hw.logicalcpu)
forge run --log-level debug を実行して行ごとの評価トレースを出力します。 ファイルにリダイレクト（2> debug.log）することで、ターミナルを汚さずに 完全な評価タイムラインを確認できます。

Overview

Architecture

AlphaForge operates as a three-stage pipeline: ingestion, descriptor evaluation, and result emission. Each stage is independently configurable and testable. 

┌─────────────────────────────────────────────────────────────────┐
│                     AlphaForge Pipeline                         │
│                                                                 │
│  ┌──────────────┐   ┌──────────────────┐   ┌────────────────┐  │
│  │  forge ingest│──▶│  forge run       │──▶│  forge         │  │

The descriptor (.yaml) defines which series to watch, the baseline computation method, and the threshold conditions that constitute an anomaly. The descriptor does not contain execution logic; AlphaForge interprets it at simulation time.

Step 1 of 5

Ingest Telemetry Data

Use forge ingest to load your raw JSONL log file into AlphaForge's normalised time-series store. The command validates the timestamp field, casts numeric columns, and partitions the data by host identifier.

Step 2 of 5

Write the Descriptor

A descriptor is a declarative YAML file that tells AlphaForge which series to monitor, how to compute a rolling baseline, and what threshold conditions trigger an anomaly event. No scripting or compiled code is required.

Create the descriptor file

forge strategy create \
  --template anomaly_detector \
  --id resource_spike_detector \
  --out descriptors/resource_spike_detector.yaml

Open the generated file and configure it as follows. Every field is documented inline.

descriptors/resource_spike_detector.yaml

# AlphaForge Descriptor — Server Resource Spike Detector
# Version schema: https://alforge-labs.github.io/schema/descriptor/v1
id: resource_spike_detector
version: "1.0.0"
description: >
  Detects sustained CPU, memory, and network-latency spikes in per-host
  server telemetry. Emits a structured anomaly event whenever a series
  exceeds its rolling baseline by the configured sigma multiplier for at
  least the specified minimum duration.

# ── Input ────────────────────────────────────────────────────────────────
input:
  partition_field: host          # Group observations by this field
  timestamp_field: ts

# ── Series Definitions ───────────────────────────────────────────────────
series:
  - name: cpu_pct
    unit: percent                # Informational; not used in computation
    window: 10m                  # Rolling window for baseline computation
    baseline: rolling_mean       # Options: rolling_mean | rolling_median | rolling_percentile

  - name: mem_pct
    unit: percent
    window: 10m
    baseline: rolling_mean

  - name: net_latency_ms
    unit: milliseconds
    window: 5m
    baseline: rolling_percentile
    percentile: 95               # Required when baseline: rolling_percentile

# ── Anomaly Thresholds ───────────────────────────────────────────────────
thresholds:
  cpu_pct:
    sigma: 3.0                   # Standard deviations above baseline
    min_duration: 2m             # Must persist this long to emit an event

  mem_pct:
    sigma: 2.5
    min_duration: 5m

  net_latency_ms:
    sigma: 4.0
    min_duration: 30s

# ── Output ───────────────────────────────────────────────────────────────
output:
  format: jsonl                  # Options: jsonl | csv | parquet
  path: ./output/anomalies.jsonl
  include_context_window: true   # Append pre/post-event observations to each record
  context_window: 5m             # Lookback/lookahead duration

Register the descriptor

forge strategy save descriptors/resource_spike_detector.yaml

# Output:
# ✔  Registered  resource_spike_detector  (v1.0.0)
Baseline options at a glance:
rolling_mean — arithmetic mean of the window; sensitive to isolated spikes.
rolling_median — robust to outliers; slower to reflect sustained shifts.
rolling_percentile — use for long-tail distributions (e.g. latency).

Step 4 of 5

Run the Simulation

Execute the full simulation across the ingested time range. AlphaForge iterates over each partition, computes rolling baselines, evaluates thresholds at every observation, and emits anomaly events to the configured output path. 

forge run \
  --strategy resource_spike_detector \
  --data ./data/series/ \
  --from 2026-01-15T00:00:00Z \
  --to   2026-01-15T23:59:59Z \
  --output ./output/ \
  --workers 4

Troubleshooting

  • No anomaly events emitted despite visible spikes in source data

    The most common cause is a min_duration that exceeds the actual spike duration in the data. Reduce min_duration (e.g. from 5m to 30s) and re-run. Also verify the rolling window is not wider than the total observation range, which leaves too few rows to compute a meaningful baseline. 

    # Inspect the rolling baseline for a specific series and partition
forge debug baseline \
  --strategy resource_spike_detector \
  --series cpu_pct \
  --partition web-01 \
  --output ./debug/
  • Ingest fails with ParseError: invalid timestamp

    AlphaForge expects ISO-8601 UTC timestamps (e.g. 2026-01-15T08:00:00Z) or Unix epoch in milliseconds. If your source uses a different format, convert it before ingesting: 

    # Convert epoch-seconds to ISO-8601 using jq
jq 'del(.ts) | .ts = (.epoch_s | todate)' raw.jsonl > telemetry.jsonl
  • High memory consumption with large JSONL files

    Convert your input to Parquet before ingesting. Parquet is columnar and supports predicate pushdown, which substantially reduces the memory footprint for wide schemas: 

    # Convert JSONL to Parquet using DuckDB (no install required via uv)
uvx duckdb -c "
  COPY (SELECT * FROM read_json_auto('telemetry.jsonl'))
  TO 'telemetry.parquet' (FORMAT PARQUET);
"

forge ingest telemetry.parquet \
  --format parquet \
  --partition-field host \
  --series cpu_pct mem_pct net_latency_ms
  • Simulation is slow on large datasets

    Increase --workers to match available CPU cores. Each partition is processed independently, so parallelism scales linearly up to the partition count: 

    forge run \
  --strategy resource_spike_detector \
  --workers $(nproc) \    # Linux
  --output ./output/

# macOS equivalent
forge run --strategy resource_spike_detector --workers $(sysctl -n hw.logicalcpu)
Run forge run --log-level debug to emit per-row evaluation traces. Redirect to a file (2> debug.log) to inspect the full evaluation timeline without cluttering your terminal.
